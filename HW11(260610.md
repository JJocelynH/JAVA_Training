1. Why spring boots? pros and cons
	**Pros:**
	
	- **Auto-configuration** — automatically configures your app based on the dependencies you added, no need for verbose XML or manual Bean setup
	- **Embedded server** — comes with built-in Tomcat, so you just run `java -jar`, no separate server deployment needed
	- **Starter dependencies** — one starter manages a group of related dependencies with compatible versions
	- **Production-ready** — built-in Actuator for health checks and monitoring out of the box
	- **Rich ecosystem** — seamlessly integrates with Spring Security, Spring Data, Spring Cloud
	
	**Cons:**
	
	- **Magic black box** — auto-configuration happens behind the scenes, making it harder to debug when things go wrong
	- **Large JAR size** — packages many dependencies you might not actually use
	- **Slower startup** — compared to lightweight frameworks like Micronaut or Quarkus
	- **Higher memory footprint** — not ideal for resource-constrained environments
    
1. How to start a spring boot project from scratch?

To start a Spring Boot project from scratch, you would first go to **start.spring.io**, which is Spring's official project generator. There you select your build tool — usually Maven — your Java version, and add the dependencies you need, like Spring Web or Spring Data JPA. Then you download and unzip the generated project.

Next you open it in IntelliJ, which will automatically recognize it as a Maven project and pull in all the dependencies for you.

Once it's open, you'll see the main class annotated with `@SpringBootApplication` — that's the entry point. You just run that class and Spring Boot starts up an embedded Tomcat server on port 8080 by default.

From there you can start building — create your controllers, services, and repositories, and configure your database connection in `application.properties`.
    
2. @Controller vs @RestController

`@RestController` = `@Controller` + `@ResponseBody`.

`@Controller` returns a **view name** — Spring looks for an HTML template to render.

`@RestController` automatically serializes the return value to **JSON** and writes it to the response body directly.

Use `@Controller` for server-side rendered pages, `@RestController` for REST APIs.
    
3. @PathVariable vs @RequestParam

`@PathVariable` extracts a value directly from the URL path. `@RequestParam` extracts a value from the query string after the question mark. In general, you use `@PathVariable` when you want to identify a specific resource, and `@RequestParam` when you want to filter or search.
    
4. @RequestBody vs @ResponseBody

`@RequestBody` is used on the method parameter — it tells Spring to take the incoming HTTP request body and deserialize it from JSON into a Java object. You typically use this in POST or PUT requests when the client is sending data.

`@ResponseBody` is used on the method itself — it tells Spring to take the return value, serialize it to JSON, and write it directly to the HTTP response body instead of looking for a view to render.
    
5. how to use GetMapping, PutMapping, PostMapping, DeleteMapping, RequestMapping?

These annotations are used to map HTTP requests to specific methods in your controller. `@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping` are just shortcuts that have the HTTP method already baked in, so you don't need to specify it manually.
In practice, you put `@RequestMapping("/users")` on the class to set the base path, then use `@GetMapping` to fetch data, `@PostMapping` to create a new resource, `@PutMapping` to update an existing one, and `@DeleteMapping` to delete it. Each method annotation takes a path like `"/{id}"` if you need to target a specific resource.
    
6. What is Spring Actuator?

Spring Actuator is a Spring Boot built-in tool that exposes monitoring endpoints automatically once you add the dependency. The most commonly used ones are `/actuator/health` to check if the app is up, `/actuator/metrics` to see things like memory usage and request counts, and `/actuator/env` to inspect environment variables. In production, it integrates with Prometheus to scrape those metrics and Grafana to visualize them as dashboards.
    
7. how to achieve async in spring boot application
In Spring Boot, you achieve async by adding `@EnableAsync` on your main class or configuration class to turn on async support, and then annotating the method you want to run asynchronously with `@Async`. That method will then run in a separate thread instead of blocking the caller.

If you want the caller to get the result eventually, the method should return a `CompletableFuture` instead of a plain value. If you don't need a return value at all, just return `void`.

You can also define a custom thread pool by configuring a `TaskExecutor` bean, which lets you control things like the pool size and queue capacity instead of using Spring's default thread pool.
    
8. how does spring handle exception

Spring handles exceptions through `@ExceptionHandler` and `@ControllerAdvice`.

`@ExceptionHandler` is defined inside a controller and handles exceptions thrown by methods in that specific controller. 
But if you want to handle exceptions globally across all controllers, you use `@ControllerAdvice` or `@RestControllerAdvice` for REST APIs — which acts as a centralized exception handler for the whole application.

Inside the advice class, you define methods annotated with `@ExceptionHandler` for each exception type you want to catch, and return a meaningful response with an appropriate HTTP status code. 
    
9. how does spring validate data

Spring validates data using the `@Valid` annotation together with Bean Validation annotations.

You put validation annotations directly on your model fields.

Then in your controller, you put `@Valid` on the `@RequestBody` parameter, and Spring will automatically validate the incoming data before it even reaches your service layer.
    
10. how does spring do the log

Spring handles logging through AOP. You define an `@Aspect` class with `@Before` or `@Around` advice to intercept method calls and log automatically, so you don't have to manually add log statements in every method. This keeps logging logic separated from your business logic.
    
11. cache hit vs cache miss
A cache hit is when the requested data is found in the cache, so the system returns it directly without hitting the database, which is fast. A cache miss is when the data is not in the cache, so the system has to go to the database to fetch it, store it in the cache, and then return it, which is slower. The goal is to maximize cache hits to reduce database load and improve response time.
    
12. do some research on Redis and have a basic understanding
**What is Redis?**

Redis stands for Remote Dictionary Server. It is an open-source, in-memory data structure store used as a database, cache, message broker, and streaming engine. It stores all data in RAM for ultra-fast access, with optional persistence to disk. [Medium](https://medium.com/@smita.s.kothari/complete-guide-to-redis-integration-use-cases-and-spring-boot-examples-99853c71565b)

**Data Structures it supports**

Redis supports strings, lists, hashes, sets, sorted sets, bitmaps, and streams. Each has specific use cases — for example, strings for caching frequently accessed data, lists for message queues, hashes for storing user profiles, and sorted sets for leaderboards and rankings. 

**Common Use Cases**

Redis is particularly popular for caching, session management, and real-time analytics — anywhere quick data retrieval is essential. [

**How it works in Spring Boot**

You add the `spring-boot-starter-data-redis` dependency, then use `@Cacheable` on your service methods. The first call fetches from the database; subsequent calls return directly from Redis cache. You can also configure TTL so cached data expires automatically. 

**Key characteristics**

Redis uses a single-threaded model but is extremely fast because there is no context switching or locking overhead. It also supports horizontal scaling through Redis Cluster and sharding.

    
13. @RestControllerAdvice instead of @ControllerAdvice

`@RestControllerAdvice` is just `@ControllerAdvice` + `@ResponseBody`, the same relationship as `@RestController` vs `@Controller`.

So `@ControllerAdvice` is for traditional web apps that return HTML views, while `@RestControllerAdvice` is for REST APIs where you want to return error responses as JSON directly. In practice, since you're always building REST APIs, you always use `@RestControllerAdvice`.

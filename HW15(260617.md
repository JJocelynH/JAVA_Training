1. jdbc vs hibernate
		jdbc is a low level java api where you write raw SQL yourself and manually handle everything, for example opening connections, setting parameters etc. Hibernate is an ORM that sits on top of JDBC and maps java objects to database tables automatically, so you work with objects instead of SQL and it generates the queries for you
		The tradeoff is control vs convenience. JDBC gives you full control over what SQL runs, which makes it easier to optimize. HIbernate eliminates boilerplate and handles relationships and caching nicely, but generated SQL can be inefficient and harder to debug.    
		
2. Statement vs PreparedStatement vs CallableStatement
		All three are JDBC interfaces for executing SQL.
		Statement is the simplest, you just pass a raw SQL string directly. The problem is you have to concatenate values into the string yourself. It will opens you up to SQL injection and means the database reparses and recompiles the query every time it runs.
		PreparedStatement solves both problems. You write the SQL once with ? placeholders, and the database pre-compiles it. Then set values saparately with typed setter like setInt() or setString(). This makes it safe from injection, more readable and faster when the same query runs multiple times since the execution plan is cached
		CallableStatement extends PreparedStatement for calling stored procedures in the database.It supports both IN parameters (values you pass in) and OUT parameters (values the procedure returns back to you),
    
3. how to prevent sql injection
	The main way is to use **PreparedStatement** with `?` placeholders instead of concatenating user input into the SQL string directly. The database treats the parameter values as pure data, never as executable SQL, so even if someone passes in malicious input it just gets treated as a literal string.
	
	Do Input validation to reject input that doesn't match what you expect.
	
	**Least privilege** — the database user your app connects with should only have the permissions it actually needs. If it only needs to read data, don't give it DELETE or DROP rights. That way even a successful injection can't do as much damage.
	
	**ORM** — using Hibernate or Spring Data JPA handles parameterization for you automatically, so you get the same protection without thinking about it manually.
    
1. what is ORM
	ORM stands for object-relational mapping. It's a technique that lets you work with database data as Java objects instead of writing raw SQL. The core idea is your database hans tables and rows, your java code has classes and objects. ORM maps between the two automatically.
	Hibernate is the most common ORM framework in Java, and Spring Data JPA builds on top of it to make things even simpler.
    
2. JPA vs Hibernate
	JPA（java persistence api) is a specification. It is a set of rules and interfaces that define how ORM should work in Java. It defines annotations like @Entity, @Id, @OneToMany, and interfaces like EntityManager, without saying how they're implemented under the hood.
	Hibernate is one of several implementations of that specification. It's the actual code that does the work. When you use @Entity form JPA, Hibernate is the engine that reads that annotation and generates the real SQL to make it happen.

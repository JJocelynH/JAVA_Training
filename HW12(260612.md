***what Spring annotations have you used***
I've used annotations across a few categories. For IOC, `@SpringBootApplication` is the main one — it combines `@EnableAutoConfiguration`, `@Configuration`, and `@ComponentScan`, which scans for `@Controller`, `@RestController`, `@Service`, `@Repository`, `@Component`, and `@Bean`. I've also used `@Scope` to control bean scope like singleton or prototype.

For dependency injection, I've used `@Autowired` with constructor, field, and setter injection, and `@Qualifier` when there's a type conflict between multiple beans of the same type.

For AOP, I've used `@Aspect` with `@Pointcut` to define where and when advice runs, along with `@Before`, `@After`, and `@Around`. I've also used `@RestControllerAdvice` to handle exceptions globally across controllers.

For REST API design, `@RestController` combines `@Controller` and `@ResponseBody`. I've used `@RequestMapping` along with `@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping` for endpoints, and `@RequestParam`, `@PathVariable`, `@RequestHeader`, and `@RequestBody` for handling input. For responses, I use `@ResponseBody` along with appropriate HTTP status codes.


***how does spring ioc work?***
Spring IOC works by managing the creation, configuration, and lifecycle of objects called beans, instead of you creating them with `new`.

It starts at `@SpringBootApplication`, which combines `@EnableAutoConfiguration`, `@Configuration`, and `@ComponentScan`. `@ComponentScan` scans your packages for classes annotated with `@Component`, `@Service`, `@Repository`, `@Controller`, or `@RestController`, and registers them as beans in the IOC container. You can also manually register beans using `@Bean` methods inside a `@Configuration` class.

Once beans are registered, Spring handles dependency injection — it looks for `@Autowired` annotations on constructors, fields, or setters, finds matching beans by type, and injects them automatically. If there's a type conflict between multiple beans, `@Qualifier` is used to specify which bean by name.

By default every bean is a singleton — Spring creates one instance and shares it everywhere. You can change this with `@Scope` to make a bean prototype, request, or session scoped depending on the use case.

Spring also manages the bean lifecycle — `@PostConstruct` runs right after a bean is created and its dependencies are injected, and `@PreDestroy` runs right before the bean is destroyed on app shutdown.

All of this is managed by the `ApplicationContext`, Spring's IOC container, which eagerly creates all singleton beans at startup and wires everything together based on these annotations.

***why aop***
AOP stands for Aspect Oriented Programming. It allows you to add extra functionality around your original logic without modifying it directly. Common use cases include logging, security checks, and timers.

Logging — automatically log when a method starts, ends, or throws an exception, without writing that logging code in every method.

Security checks — verify if a user has permission to call a method before it executes, useful for role-based access control.

Timers — measure how long a method takes to execute, useful for performance monitoring.

Transaction management — automatically start a database transaction before a method runs and commit or rollback after, depending on whether it succeeds or throws an exception.

Caching — check if a result is already cached before running the method, and store the result after, so repeated calls with the same input don't redo expensive work.

Exception handling — catch and handle exceptions in a centralized way across multiple methods, often combined with `@RestControllerAdvice`.

***how do you implement exception handling in Spring Boot***
There are two parts — implementation and handling. On the implementation side, Java exceptions are split into runtime (unchecked) exceptions, which don't need to be declared or caught, and compile-time (checked) exceptions, which the compiler forces you to either catch or declare with `throws`.

For handling, Spring uses AOP under the hood. You can use `@RestControllerAdvice` combined with `@ExceptionHandler` to catch exceptions globally across all controllers and return a consistent error response. Alternatively you can use `@Aspect` with `@Pointcut` and advice types like `@Around`, `@Before`, `@After`, `@AfterThrowing`, or `@AfterReturning` to intercept methods and handle exceptions at specific points.

`@AfterThrowing` specifically runs when a method throws an exception, letting you log it or trigger some cleanup logic, while `@RestControllerAdvice` is the more common and cleaner approach for REST APIs since it centralizes error responses with proper HTTP status codes.

constructor injection is preferred because it prevents `NullPointerException`, since dependencies are guaranteed to be set when the object is created. It's also easier for testing, because you can pass in mock dependencies directly through the constructor without needing Spring's container at all. how can you use profiles
You can set the active profile in `application.properties` or `application.yml` with `spring.profiles.active=dev`. In YAML it would look like:

```yaml
spring:
  profiles:
    active: dev
```

You can also set it via command line when running the jar: `java -jar springboot.jar --spring.profiles.active=qa`.

Another way is through environment variables — Spring Boot loads `application.properties` first, then a profile-specific file like `application-qa.properties` on top of it, overriding any matching values. On a remote Linux server, you'd set this with `export SPRING_PROFILES_ACTIVE=qa` in the terminal before starting the app.

Profiles let you have different configurations for different environments — dev, qa, prod — without changing your code.




**What is auto configuration?**

`@EnableAutoConfiguration` tells Spring Boot to automatically configure beans based on what's on your classpath. For example, if it sees a database driver, it auto-configures a `DataSource` bean. If it sees Tomcat, it sets up an embedded web server. This is what lets Spring Boot apps run with minimal configuration.

**How to stop auto configuration?**

You exclude specific auto-configuration classes using `@SpringBootApplication(exclude = {SomeAutoConfiguration.class})`. The example shown, `exclude = com.google.model.*`, isn't quite right syntax-wise — you need to point to actual auto-configuration classes, not a package wildcard.

**Advantages of Spring Boot Framework**

Spring Boot is built on top of Spring, so you get IOC and AOP for free. On top of that, it adds auto-configuration so you don't have to manually wire everything, an embedded Tomcat server so you don't need to deploy to an external server, starter dependencies that bundle related libraries together, and production-ready features like Actuator for monitoring health and metrics.

**Tomcat, autoconfiguration, actuator, properties/yaml**

Spring Boot comes with an embedded Tomcat server, so your app runs as a standalone jar without needing a separate server installation. Auto-configuration reduces boilerplate setup. Actuator exposes endpoints like `/health` and `/metrics` for monitoring your app in production. Configuration can be written in `.properties` or `.yaml` format — `.yaml` is more readable for nested configuration.

**Boot starter package**

Spring Boot starters are dependency bundles you add in `pom.xml`, like `spring-boot-starter-web` which brings in everything you need for building a web app — Spring MVC, embedded Tomcat, Jackson for JSON — so you don't have to manually add each dependency and worry about version conflicts.



how do you define profile

You define a profile by creating separate configuration files for each environment, like `application-dev.properties`, `application-qa.properties`, and `application-prod.properties`. Each file holds environment-specific settings, like different database URLs or logging levels.

You can also mark a bean as belonging to a specific profile using `@Profile("dev")` on a `@Component` or `@Bean`, so that bean only gets created when that profile is active.

Then you activate a profile using `spring.profiles.active` in `application.properties`, as a command-line argument, or as an environment variable.


what discovery service impl you used before
Service discovery solves the problem of microservices needing to find each other's network location when that location can change — services get restarted, scaled up or down, or moved to different servers, so hardcoding IP addresses doesn't work.

With Eureka, each microservice registers itself with the Eureka server when it starts up, sending its service name and network address. The Eureka server keeps this registry up to date through periodic heartbeats — if a service stops sending heartbeats, Eureka removes it from the registry after a timeout.

When one service needs to call another, instead of using a hardcoded URL, it asks Eureka for the current location of that service by name, and Eureka returns the available instances. This is often combined with a load balancer so requests get distributed across multiple instances of the same service.

This setup is what makes it possible to scale services up and down dynamically without manually updating configuration everywhere.


**how to write spring boot to call from frontend to backend and save data to database**
Here's the flow:



The frontend sends an HTTP POST request with JSON data to a controller endpoint. The controller, annotated with `@RestController`, uses `@RequestBody` to automatically convert that incoming JSON into a Java object.

The controller passes this object to the service layer, which contains the business logic and decides what to do with the data.

The service calls the repository layer, which extends `JpaRepository`. Calling `save()` on the repository automatically generates the SQL insert statement and writes the data to the database — you don't need to write any SQL yourself.

The entity class represents the database table, with `@Entity` marking it as a table and `@Id` marking the primary key, which is usually auto-generated.

Once saved, the response — typically the saved object including its newly generated id — is sent back to the frontend as JSON.




Spring boot actuator

Spring Boot Actuator is a module that gives you production-ready monitoring out of the box — things like health checks and performance metrics — exposed as HTTP endpoints without you having to write any of that code yourself.

To use it, you just add the `spring-boot-starter-actuator` dependency to your `pom.xml`, and that alone gets it working. By default though, only the health endpoint is exposed for security reasons, so you need to configure which endpoints you actually want to show in your `application.properties`. Common ones people expose are health for app status, metrics for things like memory and CPU usage, info for app metadata, and env for environment variables. You can list them out individually, but it's important not to expose everything to external users — endpoints like env or beans can leak sensitive information like database credentials or internal configuration, so in production you'd typically restrict access to these endpoints, often only exposing them on an internal network or behind authentication.

If you want to integrate with Prometheus and Grafana, you add another dependency called micrometer-registry-prometheus, which formats Actuator's metrics in a way Prometheus can read and automatically creates a Prometheus endpoint. Prometheus then periodically scrapes that endpoint and stores the metrics over time. Finally, Grafana connects to Prometheus as a data source and turns those metrics into dashboards, so you can visually see things like request counts, response times, and memory usage over time.

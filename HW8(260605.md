- write out the optimized Singleton Version and explain each line of code
  ```java
  public class Singleton {
    
    // 1. The single instance, volatile ensures visibility across threads
    private static volatile Singleton instance;
    
    // 2. Private constructor prevents external instantiation
    private Singleton() {}
    
    // 3. Public method to get the single instance
    public static Singleton getInstance() {
        
        // 4. First check (no locking) for performance
        if (instance == null) {
            
            // 5. Synchronize only when instance might be null
            synchronized (Singleton.class) {
                
                // 6. Second check inside lock to prevent duplicate creation
                if (instance == null) {
                    instance = new Singleton();
                }
            }
        }
        return instance;
    }
}
  ```  
`volatile` → guarantees that changes to `instance` are visible to all threads immediately, prevents CPU caching issues.

`private Singleton()` → blocks anyone from doing `new Singleton()` outside this class.

**First `if (instance == null)`** → avoids locking every time, once instance is created this check fails immediately and returns — better performance.

`synchronized` → only one thread can enter this block at a time, prevents two threads creating two instances simultaneously.

**Second `if (instance == null)`** → necessary because two threads could both pass the first check before either creates the instance. The second check ensures only one actually creates it.

This pattern is called **Double-Checked Locking** — the most optimized thread-safe Singleton implementation in Java.

- **500 Internal Server Error** → something crashed on the server side, not your fault

1. TCP 3 way handshaking

TCP 3-way handshake is the process of establishing a reliable connection before any data is sent. The client starts by sending a **SYN** ("I want to connect"), the server responds with **SYN-ACK** ("OK, I'm ready"), and the client confirms with **ACK** ("let's go") — after these three steps, data transfer begins. This is why TCP is slower but more reliable than UDP, which just sends data without any handshaking.
    
2. TCP vs UDP

**TCP** (Transmission Control Protocol) is connection-oriented — it requires a 3-way handshake before sending data, guarantees delivery, ensures packets arrive in order, and retransmits lost packets. This makes it reliable but slower. Used for HTTP, emails, file transfers — anything where accuracy matters.

**UDP** (User Datagram Protocol) is connectionless — it just sends data without any handshaking, no guarantee of delivery, no order, no retransmission. This makes it fast but unreliable. Used for video streaming, gaming, live calls — anything where speed matters more than perfection.
    
3. what is Tomcat

Tomcat is a **web server** that runs Java web applications. It listens for incoming HTTP requests, passes them to your Spring Boot app, and sends the response back to the client.

Without a server, your Java code has no way to receive HTTP requests from the outside world.

**In Spring Boot:** Tomcat is embedded by default — meaning it's bundled inside your app, so you don't need to install or configure it separately. That's why you can just run your Spring Boot app and immediately hit `localhost:8080`.
    
4. what are the basic components for tomcat

**1. Connector** Listens for incoming HTTP requests on a port (default 8080) and passes them into Tomcat.

**2. Engine** The core processing component — receives the request from the Connector and figures out which application should handle it.

**3. Host** Represents a virtual host (like `localhost`) — maps requests to the correct web application.

**4. Context** Represents a single web application — maps the URL path to the correct servlet.

**5. Servlet** The actual Java component that processes the request and generates the response. In Spring, `DispatcherServlet` is the main servlet that routes requests to your controllers.

**Request flow:**

```
HTTP Request → Connector → Engine → Host → Context → Servlet → Your Controller
```
    
5. what is web server

A web server is a software that listens for incoming HTTP requests and sends back responses. When a client sends a request, the web server receives it, processes it, and returns the appropriate response, which could be HTML, JSON, or files. Examples of web servers include Tomcat, Nginx, and Apache. The difference between a web server and an application server is that a web server typically handles static content like HTML and images, while an application server handles dynamic content including business logic and database interactions — though Tomcat is capable of doing both.
    
6. what is 3 tire architecture

3-tier architecture is a software design pattern that separates an application into three distinct layers. The first layer is the **Presentation Tier**, which is the frontend that the user interacts with, such as a browser or mobile app — this is responsible for displaying data and sending user requests. The second layer is the **Application Tier** (also called Business Logic Tier), which is the backend that processes requests, applies business rules, and coordinates data flow — this is where your Spring Boot controllers and services live. The third layer is the **Data Tier**, which is the database that stores and retrieves data — this is where your PostgreSQL and JPA repositories come in.
    
7. what is OSI Model, what is each layer doing
The OSI (Open Systems Interconnection) model is a conceptual framework that standardizes how data is transmitted over a network, divided into 7 layers, each with a specific responsibility.

**Layer 7 - Application** → the layer closest to the user, handles high-level protocols like HTTP, FTP, SMTP. This is where your Spring Boot REST API lives.

**Layer 6 - Presentation** → responsible for data formatting, encryption, and compression. For example, converting data to JSON or SSL/TLS encryption for HTTPS.

**Layer 5 - Session** → manages and maintains connections between two devices, handles opening, closing, and synchronizing sessions.

**Layer 4 - Transport** → responsible for end-to-end communication, breaks data into segments, and ensures reliable delivery. This is where **TCP and UDP** operate.

**Layer 3 - Network** → handles routing and IP addressing, determines the best path for data to travel across networks. This is where **IP** operates.

**Layer 2 - Data Link** → handles communication between two directly connected devices, packages data into frames, and handles MAC addresses.

**Layer 1 - Physical** → the actual physical hardware — cables, switches, wireless signals, converts data into bits (0s and 1s).

11. # Autowire in Spring
	**1. Coding to Interface** Replace concrete classes (`Laptop`/`Desktop`) with an interface (`Computer`) for better flexibility and modularity.

	**2. Manual vs Automatic Injection**
	
	- Manual: explicitly define dependencies in `spring.xml` using `<property ref="..."/>`
	- Automatic: use the `autowire` attribute and let Spring find and inject the matching bean

	- `byName` → matches bean by variable name, requires the variable name and bean id to match

16. Spring web PUT and Delete
**1. PUT (Update)**

- Use `@PutMapping` + `@RequestBody` to receive updated data
- Since data is stored in a List (not a database), manually loop through to find the matching item by index and replace it
- Common mistake: sending plain text instead of JSON → causes `Unsupported Media Type` error, always set `Content-Type: application/json` in Postman

**2. DELETE**

- Use `@DeleteMapping` + `@PathVariable` to get the ID from the URL, e.g. `/products/1`
- Manually loop through the list to find the item and call `.remove()` to delete it

 The manual loop approach is just for learning purposes to understand the logic. In real projects (like your ems-backend), **Spring Data JPA** handles all of this automatically — no manual looping needed, just call `repository.save()`, `repository.findById()`, `repository.deleteById()`.

***what Spring annotations have you used***
I've used annotations across a few categories. For IOC, `@SpringBootApplication` is the main one — it combines `@EnableAutoConfiguration`, `@Configuration`, and `@ComponentScan`, which scans for `@Controller`, `@RestController`, `@Service`, `@Repository`, `@Component`, and `@Bean`. I've also used `@Scope` to control bean scope like singleton or prototype.

For dependency injection, I've used `@Autowired` with constructor, field, and setter injection, and `@Qualifier` when there's a type conflict between multiple beans of the same type.

For AOP, I've used `@Aspect` with `@Pointcut` to define where and when advice runs, along with `@Before`, `@After`, and `@Around`. I've also used `@RestControllerAdvice` to handle exceptions globally across controllers.

For REST API design, `@RestController` combines `@Controller` and `@ResponseBody`. I've used `@RequestMapping` along with `@GetMapping`, `@PostMapping`, `@PutMapping`, and `@DeleteMapping` for endpoints, and `@RequestParam`, `@PathVariable`, `@RequestHeader`, and `@RequestBody` for handling input. For responses, I use `@ResponseBody` along with appropriate HTTP status codes.


***how does spring ioc work?***
Spring IOC works by managing the creation, configuration, and lifecycle of objects called beans, instead of you creating them with `new`.

It starts at `@SpringBootApplication`, which combines `@EnableAutoConfiguration`, `@Configuration`, and `@ComponentScan`. `@ComponentScan` scans your packages for classes annotated with `@Component`, `@Service`, `@Repository`, `@Controller`, or `@RestController`, and registers them as beans in the IOC container. You can also manually register beans using `@Bean` methods inside a `@Configuration` class.

Once beans are registered, Spring handles dependency injection — it looks for `@Autowired` annotations on constructors, fields, or setters, finds matching beans by type, and injects them automatically. If there's a type conflict between multiple beans, `@Qualifier` is used to specify which bean by name.

By default every bean is a singleton — Spring creates one instance and shares it everywhere. You can change this with `@Scope` to make a bean prototype, request, or session scoped depending on the use case.

Spring also manages the bean lifecycle — `@PostConstruct` runs right after a bean is created and its dependencies are injected, and `@PreDestroy` runs right before the bean is destroyed on app shutdown.

All of this is managed by the `ApplicationContext`, Spring's IOC container, which eagerly creates all singleton beans at startup and wires everything together based on these annotations.

***what Spring Boot version did you use***
I used Spring Boot 3.0, which requires Java 17 or higher. One important change with Spring Boot 3 is the namespace migration from `javax` to `jakarta` — so things like `javax.servlet` and `javax.persistence` became `jakarta.servlet` and `jakarta.persistence`. This matters if you're working with an older Spring Boot 2 project, since all those imports would need to be updated when migrating to Spring Boot 3.

1. what is Spring IOC
 Spring IOC, you flip that around. You just declare what you need, and Spring creates the objects, wires them together, and manages their whole lifecycle for you. That's the "inversion" — the control shifts from you to the framework.
    
2. what is IOC Container
It's a Spring component that reads your configuration, creates your beans, and manages their lifecycle.
    
3. what is the advantage come with IOC
	1. loose coupling: your classes just declare what they need and spring will manage the objects
	2. less boilerplate: spring handle the object creation and lifecycle so you don't need to do all the setup code by yourself
	3. easy to test: because the dependencies are injected, you can easily swap in mock objects during test
	4. centralized configuration: because all your objects are defined in one place, your code become easy to read and maintain
	5. easy singleton management: spring handles all the objects for you so you don't have to implement patterns like singleton by yourself
    
4. what is Dependency Injection (DI)
		DI is an mechanism that spring use to implement IOC. Instead of classes create dependencies themselves, spring injects dependencies from the outside. So classes just specify what they need and spring will find the right bean and hand it in.

6. what are different types of Dependency Injection
	1. constructor injection: dependencies pass through the constructor and this is the most recommended way
	2. setter injection:  Spring call setter method to inject dependencies after the object is created
	3. field injection: use @Autowired directly on the field, it is the least recommended because it makes the testing harder
    
7. what are the pros and cons for each types of dependency Injection
	1. **Constructor injection** is the most recommended. Your dependencies are set at object creation and can be `final`, so the object is always in a valid state. The downside is it can get verbose if you have many dependencies
	2. **Setter injection** is the shortest to write, but Spring injects it using reflection behind the scenes, which makes it hard to test and hides what your class actually depends on. Most teams avoid it in production code.
	3. **Field injection** is the shortest to write, but Spring injects it using reflection behind the scenes, which makes it hard to test and hides what your class actually depends on. Most teams avoid it in production code.
    
8. @Component vs @Bean
	Both register a bean into the IOC container, but they're used in different situations.
	
	`@Component` goes on the class itself, and Spring auto-detects it through component scanning. You use it for your own classes that you wrote.
	
	`@Bean` goes on a method inside a `@Configuration` class, and you manually write the code to create the object. You use it when you don't own the class — for example, a third-party library class that you can't add `@Component` to.
    
9. what is @Configuration and @ComponentScan
	`@Configuration` marks a class as a configuration class — it's where you define `@Bean` methods. Spring treats it as a source of bean definitions.

	`@ComponentScan` tells Spring which package to scan for `@Component` classes. Without it, Spring doesn't know where to look. The good news is `@SpringBootApplication` already includes both, so in a Spring Boot project you rarely need to write them yourself.
    
10. @Controller vs @RestController

	Both are used for web layer classes that handle HTTP requests, but they differ in how they return data.
	
	`@Controller` is meant for returning views — like HTML pages. It works with a template engine like Thymeleaf.
	
	`@RestController` is `@Controller` + `@ResponseBody` combined. It means every method automatically returns data as JSON instead of a view. This is what you use when building a REST API.
    
11. @Controller vs @Service vs @Repository
	
	All three are just specialized versions of `@Component`, so they all register a bean. The difference is semantic — they tell you which layer the class belongs to.
	
	`@Controller` is the web layer, it handles incoming HTTP requests.
	
	`@Service` is the business logic layer, it processes the actual work.
	
	`@Repository` is the data layer, it talks to the database. It also gets extra behavior from Spring — database exceptions are automatically translated into Spring's own exception types.
    
11. spring bean scope

	Bean scope controls how many instances Spring creates for a bean.
	
	**Singleton** is the default. Spring creates one instance and shares it everywhere. Every time you ask for that bean, you get the same object.
	
	**Prototype** is the opposite. Spring creates a new instance every time you ask for that bean.
	
	There are also web-specific scopes: **Request** creates a new instance per HTTP request, and **Session** creates one per user session. But you only use those in web applications.
	
12. singleton vs prototype

	Singleton means Spring creates the bean once and shares that same instance everywhere in the app.
	
	Prototype means every time you request the bean, Spring creates a brand new instance. Spring also doesn't manage its lifecycle after handing it to you.
    
13. give me 3 uses cases for each of singleton, prototype, request and session bean scope
	**Singleton**: service classes like `UserService`, repository classes like `UserRepository`, and utility classes like a password encoder. Things that are stateless and safe to share.
	
	**Prototype**: objects that hold user-specific state, a builder object that accumulates data before creating something, and batch processing tasks where each job needs its own fresh instance.
	
	**Request**: storing data about the current HTTP request like request metadata, tracking what a user did in a single request for logging, and holding form data being processed for that one request.
	
	**Session**: storing a logged-in user's info like their name and role, keeping track of a shopping cart, and remembering user preferences like language or theme across multiple requests.
    
13. session vs cookie

	Both store user data, but in different places.
	
	Session is stored on the server. Spring holds the data in memory and gives the browser a session ID. The browser sends that ID on every request so Spring knows who you are.
	
	Cookie is stored on the browser. The actual data sits on the client side and gets sent to the server with every request.
	
	The tradeoff is that sessions are more secure because the real data never leaves the server, but they use server memory. Cookies are lightweight but less secure because the data is on the client and can be tampered with.

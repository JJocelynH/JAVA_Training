1. what is AOP

AOP lets you separate repetitive logic like logging, security, and transaction management from your business code. You write that logic once in an Aspect class, and Spring automatically applies it to whichever methods you specify via a Pointcut.

1. what is JointPoint and Aspect in AOP

A JoinPoint is the specific moment Spring intercepts — basically the method being executed. Inside your Aspect method, you can access the JoinPoint object to get info like the method name and its arguments.

An Aspect is just the class that contains all your cross-cutting logic — it's where you put your `@Before`, `@After`, or `@Around` methods.
    
2. what is Aspect, joinpoint, pointcut, advice, target

**Aspect** is the class that contains all your cross-cutting logic.
**JoinPoint** is the specific method being intercepted at runtime. You can use it to get the method name and arguments.
**Pointcut** is the expression that defines which methods to intercept.
**Advice** is the actual code that runs — the method inside your Aspect annotated with `@Before`, `@After`, or `@Around`.
**Target** is the original object whose method is being intercepted.
    
3. ApplicationContext vs BeanFactory

Both are IOC containers, but `ApplicationContext` is more feature-rich.

`BeanFactory` is the basic container — it just creates and manages beans, and it's lazy by default, meaning it only creates a bean when you ask for it.

`ApplicationContext` extends `BeanFactory` and adds extra features like event publishing, internationalization, and AOP support. It also eagerly creates all singleton beans at startup. This is what Spring Boot uses by default.

4. How is Spring MVC Working Flow 
	1. The browser sends an HTTP request, and it first hits the **DispatcherServlet** — Spring's front controller that handles everything.
	2. DispatcherServlet asks the **HandlerMapping** to figure out which controller method should handle this request.
	3. The request goes to the matching **Controller** method, which processes the business logic.
	4. The Controller returns a response — in a REST API that's just JSON data, in a traditional MVC app it returns a view name.
	5. The response goes back through DispatcherServlet and is sent to the client.

		In a `@RestController` you skip the view step entirely and just return data directly as JSON.
    
5. how does @Autowire work

When Spring starts up, it scans all your classes and registers the beans into the IOC container. When it sees `@Autowired` on a constructor, field, or setter, it looks inside the container for a bean that matches that type, and injects it automatically.

If it finds exactly one match, it injects it. If it finds zero it throws an error. If it finds multiple matches it gets confused and throws an error too, unless you use `@Qualifier` to tell it which one you want.
    
6. @Autowire and @Qualifer

`@Autowired` tells Spring to inject a bean by type. `@Qualifier` is used alongside `@Autowired` when you have multiple beans of the same type and you need to specify which one you want by name.

For example if you have two classes both implementing `PaymentService`, Spring doesn't know which one to inject. You add `@Qualifier("stripePaymentService")` to tell it exactly which bean to use.
7. 2 use cases for @PostConstruct and @PreDestroy

`@PostConstruct` runs right after the bean is created and all dependencies are injected.
Use case 1 — loading initial data into a cache right after the bean is ready. 
Use case 2 — setting up a database connection pool when the app starts up.

`@PreDestroy` runs just before the bean is destroyed when the app shuts down.
Use case 1 — closing database connections cleanly before shutdown. 
Use case 2 — saving in-memory data to disk before the app exits.

1.sql vs no sql database
SQL databases are relational, meaning data is stored in structured tables with predefined schemas and relationships between tables. They use SQL to query data and are strongly consistent, making them good for transactional systems where data integrity is critical, like banking or e-commerce. Examples are PostgreSQL, MySQL, and Oracle. NoSQL databases are non-relational and store data in flexible formats like documents, key-value pairs, wide columns, or graphs. They sacrifice some consistency for scalability and flexibility, making them better suited for large-scale distributed systems, real-time applications, or when the data structure changes frequently. Examples are MongoDB for documents, Redis for key-value, and Cassandra for wide column. The choice depends on the use case — if you need strict consistency and complex queries, use SQL. If you need high scalability and flexible schema, use NoSQL.

2. what is database normalization
Database normalization is the process of organizing a relational database to reduce data redundancy and improve data integrity by dividing large tables into smaller, related ones. It follows a series of rules called normal forms. First normal form requires that each column holds atomic values with no repeating groups. Second normal form requires that every non-key column depends on the entire primary key, not just part of it. Third normal form requires that non-key columns depend only on the primary key, not on other non-key columns. In practice normalization reduces storage and prevents update anomalies, but too much normalization can hurt read performance because you need more joins. So sometimes you deliberately denormalize for performance, which is a trade-off common in reporting or analytical systems.

 3. vertical scaling vs horizontal
Vertical scaling means adding more resources to a single server, like more CPU, more RAM, or faster storage. It's simple because you don't change the application architecture, but it has a hard limit — you can only scale up to what the hardware supports, and it's expensive. Horizontal scaling means adding more servers and distributing the load across them. It's more complex because the application needs to handle distributed state, but it's theoretically unlimited and more cost-effective at scale. In modern cloud architectures horizontal scaling is preferred because you can add or remove instances dynamically based on traffic. Stateless services like REST APIs scale horizontally very easily, while stateful services like databases are harder and require techniques like sharding or read replicas.

 4. what is ACID
ACID stands for Atomicity, Consistency, Isolation, and Durability. These are the four properties that guarantee database transactions are processed reliably. Atomicity means a transaction is all or nothing — if any part fails, the whole transaction rolls back. Consistency means a transaction brings the database from one valid state to another, never leaving it in a broken state. Isolation means concurrent transactions don't interfere with each other — each transaction sees a consistent snapshot of the data as if it were running alone. Durability means once a transaction is committed, it persists even if the system crashes, because it's written to disk. ACID properties are the foundation of relational databases and are why they're used for financial and transactional systems where data correctness is critical.

 5. what is CAP
CAP theorem states that in a distributed system, you can only guarantee two out of three properties at the same time: Consistency, Availability, and Partition tolerance. Consistency means every read gets the most recent write. Availability means every request gets a response, even if it's not the most recent data. Partition tolerance means the system continues to work even when network partitions occur and some nodes can't communicate. In a real distributed system, network partitions are unavoidable, so partition tolerance is a must. That means the real trade-off is between consistency and availability. Systems like traditional relational databases prioritize consistency over availability — they're called CP systems. Systems like Cassandra and DynamoDB prioritize availability over consistency — they're called AP systems and use eventual consistency instead.


## SQL Practice：

### （1） Create tables:
```sql
CREATE TABLE department (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100)
);

CREATE TABLE employee (
    id SERIAL PRIMARY KEY,
    name VARCHAR(100),
    department_id INT REFERENCES department(id),
    age INT,
    salary DECIMAL(10,2)
)
```

### (3) query the data

- Get only employee names and ages.
```sql
SELECT name, age
FROM employee
```
- Find employees older than 30.
```sql
SELECT *
FROM employee
WHERE age > 30
```

- Find employees whose salary is greater than 80,000.
```sql
SELECT *
FROM employee
WHERE salary > 80000
```

- List employees ordered by age (ascending).
```sql
SELECT *
FROM employee
ORDER BY age
```

- Get the top 3 highest-paid employees.
```sql
SELECT *
FROM employee
ORDER BY salary DESC
LIMIT 3
```

- Count total number of employees.
```sql
SELECT COUNT(*)
FROM employee
```

- Find the average salary of all employees.
```sql
SELECT AVG(salary) AS average_salary
FROM employee
```

- List employee name with department name.
```sql
SELECT e.name AS employee_name, d.name AS department_name
FROM employee e
JOIN department d
ON e.department_id = d.id
```

- Find employees earning the highest salary.
```sql
SELECT *
FROM employee
WHERE salary = (SELECT MAX(salary) FROM employee)
```

- Find employees earning the second highest salary.
```sql
WITH cte AS (
    SELECT *, ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn
    FROM employee
)

SELECT *
FROM cte
WHERE rn = 2
```

- Find employees earning the third highest salary.
```sql
WITH cte AS (
    SELECT *, ROW_NUMBER() OVER (ORDER BY salary DESC) AS rn
    FROM employee
)

SELECT *
FROM cte
WHERE rn = 3
```

1	how to increase young generation size in heap
To increase the young generation size in the JVM heap, you use JVM startup flags. The main ones are -Xms to set the initial heap size, -Xmx to set the maximum heap size, and -Xmn to set the size of the young generation specifically. For example, setting -Xmn to 512m means the young generation gets 512 megabytes. You can also use -XX:NewRatio to control the ratio between young and old generation. The reason you might increase young generation size is if your application creates a lot of short-lived objects — a larger young generation means minor garbage collections happen less frequently, which reduces pause times.

 2	how does spring mvc work
Spring MVC is a web framework built on the front controller pattern. Every incoming HTTP request first goes through the Spring Security filter chain for authentication and authorization. Then it reaches the DispatcherServlet, which is the front controller. DispatcherServlet uses HandlerMapping to find the correct controller method based on the HTTP method and URL combination. The controller processes the request by calling the service and repository layers, then returns a result. In a REST API with @RestController, Spring automatically serializes the return value to JSON using HttpMessageConverter and sends it back to the client.

 3	how do you handle exception in java
There are two ways. The first is try-catch-finally, where you wrap the risky code in a try block, catch specific exceptions in catch blocks, and use finally for cleanup code that always runs. The second is declaring exceptions with the throws keyword on the method signature, which tells the caller to handle it. Java has two types of exceptions: checked exceptions like IOException which the compiler forces you to handle, and unchecked exceptions which extend RuntimeException and don't require explicit handling. In Spring Boot I centralize exception handling using @RestControllerAdvice with @ExceptionHandler methods so I don't repeat try-catch blocks in every controller.

 4	rest api vs message queue
Both are ways for services to communicate, but they work differently. REST API is synchronous — the caller sends a request and waits for a response before continuing. It's simple and good for operations where you need an immediate result, like fetching user data. 
A message queue is asynchronous — the sender puts a message in the queue and moves on without waiting. The receiver processes it whenever it's ready. This decouples the two services and is good for tasks where you don't need an immediate response, like sending emails or processing reports. In practice I use REST for real-time operations and a message queue like RabbitMQ or Kafka for background tasks that shouldn't block the main flow.

 5	what annotations we use to configure customized actuator
You use @Endpoint on the class to define a custom actuator endpoint, and the id attribute becomes the URL path under slash actuator slash. Inside the class, @ReadOperation maps to a GET request, @WriteOperation maps to POST, and @DeleteOperation maps to DELETE. Spring Boot automatically registers it and exposes it under the actuator base path.

 6	can abstract class have no abstract method
Yes, an abstract class can have no abstract methods. The only requirement is that the class itself is declared with the abstract keyword, which means it cannot be instantiated directly. You might do this when you want to provide some shared implementation but still prevent direct instantiation, or when you plan to add abstract methods in the future. The key difference from an interface is that an abstract class can have state, constructors, and non-abstract methods with full implementations.

 7	how can you use optional
Optional is a Java 8 container for a value that might be null, designed to avoid NullPointerException. You create one with Optional.ofNullable for a value that might be null, Optional.of for a value you know is not null, or Optional.empty for an explicitly empty one. Then you consume it with orElse to return a default if empty, orElseThrow to throw an exception if empty, ifPresent to run code only when a value exists, or map to transform the value. The best practice is to use Optional mainly as a return type so a method can clearly signal it might not return a value, rather than returning null.

 8	What is functional interface?
A functional interface is an interface that has exactly one abstract method. It can have any number of default or static methods, but only one abstract method. The @FunctionalInterface annotation is optional but recommended — it tells the compiler to enforce this constraint. Functional interfaces are the foundation of lambda expressions in Java 8, because a lambda expression is essentially a concise implementation of that single abstract method. Common examples from the Java standard library are Runnable, which has run(), Comparator, which has compare(), and the java.util.function package which includes Function, Predicate, Consumer, and Supplier.

 9	Why do you use post, instead of put
It depends on the use case. POST is used to create a new resource where the server generates the id, and it is not idempotent — calling it multiple times creates multiple resources. PUT is used to update an existing resource at a specific URL, and it is idempotent — calling it multiple times with the same data always produces the same result. So I use POST when I'm creating something new and I don't know the id yet, like creating a new user. I use PUT when I'm updating an existing resource and I already know its id. 

 10	what is webflux? Have you used it in your project
Spring WebFlux is a non-blocking reactive web framework introduced in Spring 5, designed for high-concurrency scenarios. Traditional Spring MVC is synchronous and blocking — each request holds a thread until it completes, so threads sit idle waiting for database or API responses. WebFlux is non-blocking — threads don't wait, they handle other requests while waiting for data and continue when it arrives. WebFlux uses Mono for a single value and Flux for a stream of values. I haven't used it in production yet, but I understand it's most useful for real-time data streaming or extremely high concurrent load. For standard business applications, Spring MVC combined with Java 21 virtual threads is simpler and sufficient.

 11	What is hashmap?
HashMap is a Java data structure that stores key-value pairs and provides O(1) average time complexity for get and put operations. It works by computing a hash of the key to determine which bucket to store the entry in. When two keys hash to the same bucket, it's called a collision, and HashMap handles this using a linked list or, since Java 8, a red-black tree when the list gets long enough. HashMap allows one null key and multiple null values, and it does not guarantee any ordering of the entries. If you need ordering, use LinkedHashMap which maintains insertion order, or TreeMap which sorts by key. HashMap is not thread-safe — for concurrent access use ConcurrentHashMap.

 12	what is enable auto configuration
@EnableAutoConfiguration tells Spring Boot to automatically configure the application based on the dependencies present on the classpath. For example, if Spring Data JPA is on the classpath, it automatically sets up a datasource and entity manager. If Spring Security is present, it configures security automatically. In practice you don't add this annotation directly — it's already included inside @SpringBootApplication, which combines @EnableAutoConfiguration, @Configuration, and @ComponentScan. To exclude a specific auto-configuration, you pass it in the exclude attribute of @SpringBootApplication, or set spring.autoconfigure.exclude in application.properties.

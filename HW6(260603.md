- client - server model
The client-server model is a communication architecture. Two parties play different roles: the client initiates requests, and the server responds to them.

%% There are a few important properties of this model.
First, HTTP is stateless by default, meaning the server treats every request as independent and has no memory of previous ones. 
Second, the roles are asymmetric: clients always initiate, servers always respond (WebSockets are an exception that allows bidirectional communication). 
Third, clients and server don't need to be on different machines — during local development, both often run on your laptop. %%
    
- application service
An application service sits between the client and the database, containing the core business logic of a system. When a client sends a request, the application service receives it, validates the input, applies business rules, and decides what to read or write to the database. The client never touches the database directly.

%% This separation keeps business logic in one place, allows multiple clients to share the same logic without duplication, and protects the database from direct exposure. %%
    
- http request/response
HTTP is the protocol that defines how clients and servers communicate. The pattern is the client sends a request, the server processes it and sends back a response.

A request has three main parts. (method,url,header)
The method indicates the intended action ,GET retrieves data, POST submits data, PUT updates it, DELETE removes it. 
The URL identifies the specific resource being targeted. 
The headers carry metadata like content type, authentication tokens, or caching instructions. 
A request can also include a body, which contains the actual data being sent, though GET requests typically don't have one.

A response mirrors this structure. It contains a status code indicating the outcome —100 means  200 means success, 300 means redirection error, 400 means client side error, 500 means something went wrong on the server. It also includes headers and usually a body containing the returned data, often in JSON or HTML format.

One important characteristic is that HTTP is stateless. Each request is completely independent, and the server has no memory of previous ones. This simplifies the server's design but means that any continuity, like staying logged in, must be handled explicitly through cookies or tokens attached to each request.
    
- horizontal scaling vs vertical scaling
Vertical scaling means making a single server more powerful by adding more CPU, RAM, or storage to the existing machine. It's straightforward because your software doesn't need to change, but it has a hard ceiling: there's only so much hardware you can add to one machine.

Horizontal scaling means adding more machines and distributing the load across them. Instead of one powerful server, you run many ordinary ones in parallel. You can keep adding machines as demand grows, but it introduces complexity. Your application needs to be designed to run as multiple identical instances, sessions and state can't be stored on a single machine, and a load balancer is needed to distribute incoming requests across the pool.
    
- load balancer
A load balancer sits in front of a group of servers and distributes incoming requests across them. Its primary job is to make sure no single server gets overwhelmed while others sit idle.
Beyond distribution, a load balancer also improves availability. It continuously checks whether each server is healthy, and if one goes down, it stops sending traffic there until it recovers. 
    
- microservice, microfronted

A microservice architecture breaks a backend into small, independent services each responsible for a single business domain. Each service has its own codebase, runs as its own process, and communicates with others over the network. The tradeoff is operational complexity, instead of deploying one application, you're managing dozens of services. Each of them needs its own infrastructure, monitoring, and failure handling.

A microfrontend means that different parts of the UI are built and deployed by separate teams, then assembled together at runtime. The benefit is autonomy. However, keeping a consistent look across independently built pieces is a challenge. Performance can also suffer when each microfrontend loads its own copy of the same libraries.
    
- database: relational database (sql database), nonrelational database ( no-sql database), 

A relational database stores data in tables with rows and columns. Relationships between tables are defined through foreign keys. The structure is fixed upfront as a schema, so every record has the same shape. SQL is used to query the data. Transactions either complete fully or not at all, so the data is always in a valid state. This works well when correctness matters for example financial records, user accounts.

A non-relational database, called NoSQL, is more flexible. Instead of tables, data is stored as documents, key-value pairs, or graphs. There's no required schema, so different records can have different shapes. This makes it easier to store data that doesn't fit neatly into rows and columns.
    
- api gateway

An API gateway is a single entry point that sits in front of all backend services. Instead of clients knowing the address of each individual service, they talk only to the gateway, which routes requests to the appropriate service behind the scenes.

Beyond just routing, the gateway also takes care of things like authentication, rate limiting, logging, and caching.

The tradeoff is that the gateway becomes a critical single point of failure. This is typically addressed by running multiple instances behind a load balancer.
    
- message queue

A message queue allows services to communicate asynchronously. Instead of calling another service directly and waiting, a service places a message onto a queue and moves on. The receiver picks it up whenever it's ready.

This decouples the two services in time. If the receiver is down, messages accumulate and are processed once it recovers. 

It also helps when traffic suddenly spikes — instead of overwhelming the downstream service all at once, the queue holds the messages and lets the service work through them at its own pace.

The tradeoff is complexity: there's no immediate confirmation that something succeeded, and error handling is harder. 
    
- log, monitor

Logging is just recording everything that happens in your system. You store these records and look back at them when something goes wrong to figure out what happened and why.

Monitoring is watching your system's health in real time. Instead of reading through individual events, you're looking at big-picture metrics — CPU usage, response times, error rates — and getting alerted the moment something goes wrong.

The two work together: monitoring tells you something is wrong right now, logging helps you dig in and find out why.
    
- deployment with AWS/Azure/GCP

Cloud providers like AWS, Azure, and GCP let you run applications without managing physical hardware, renting computing resources on demand and paying for what you use.

The core part is a virtual machine — a server you can start up in minutes. Cloud providers also offer managed services, where they handle the hard parts for you. With a managed database, for example, they take care of backups, updates, and scaling. For deployment, you pack your app into a container and run it on a managed platform. The platform handles the rest — starting, stopping, and scaling your app automatically.

Cloud providers also have data centers all around the world, so you can deploy close to your users and keep things running even if one region goes down.

The downside is cost and complexity — bills can pile up fast if you're not careful, and there's a lot to learn before you know what to use and how to set it up properly.

- security (authentication and authorization)
Authentication verifies who someone is. Authorization verifies what they are allowed to do. The two are distinct — authentication answers "who are you", authorization answers "what are you allowed to do".

%% After authenticating, the server issues a token — commonly a JWT — which the client attaches to every subsequent request, allowing the server to confirm identity and check permissions without a database lookup every time.

Common vulnerabilities include stolen tokens, weak passwords, overly broad permissions, and inconsistent authorization checks across endpoints. %%
    
- why testing

Testing is how you make sure your code actually does what it's supposed to. 

It also act as a safety net when you change existing code. In any real system, touching one part can silently break another. A good test suite catches that automatically, so you can refactor or add features without being afraid of breaking something.

And also well-written tests also serve as documentation.....
